# /sim - 실패 시뮬레이션

> 완성된 코드에 `/sim`을 실행하면, "지금은 되지만 나중에 터질 곳"을 찾아냅니다.

## 언제 쓰나

- 코드를 다 짠 후 `/sim` 입력
- 리팩토링 완료 후 검증
- "이거 괜찮을까?" 싶을 때

## 3단계

### S1: 실패 시뮬레이션

변경된 코드를 Read한 뒤, "3개월 후 + 규모 10배" 상황에서 코드를 따라갑니다.

```
[추적 대상]
- 하드코딩된 값 (확장자, 모델명, 경로)
- 규모 한계 (메모리, 처리 시간, 동시 접속)
- 환경 차이 (OS, 인코딩, 권한)
- 기존 코드 충돌 (새 코드가 기존 로직을 깨뜨리는지)

[발견 시 분류]
- [P0] 블로커: 보안 구멍, 데이터 손실, 크래시
- [P1] 필수 수정: 기능 결함, 리소스 누수
- [P2] 개선: 스타일, 엣지케이스
```

규칙:
- 반드시 파일을 Read한 후 시뮬레이션 시작
- "터질 수 있다" (추상적) 금지 → "이 파일 N번째 줄에서 터진다" (구체적)

### S2: 근본 원인

- 터진 지점의 **근본 원인** 파악
- 같은 유형이 코드 어디에 더 있나 전체 탐색 (grep)
- "이것이 실제로 발생하는가?" 독립 검증 후, 통과한 항목만 보고

### S3: 수정 + 확인

```
[수정 전 검증]
S2 근본 원인 위치 == S3 수정 위치인지 확인
→ 불일치 시 수정 금지, S2 재탐색

[금지]
if X is None: return default  ← 증상 우회, 근본 원인 아님

[수정 후]
S1을 다시 1회 실행하여 확인 (최대 2회 루프)
2회 후에도 실패 → 사용자에게 보고
```

## 출력

```
[/sim 결과]
- 대상: {변경 내용}
- 읽은 파일: N개

[S1] P0: N건 / P1: N건 / P2: N건
[S2] 근본 원인 요약
[S3] 수정 N건

종합: OK / WARN / FAIL
```

## 품질 규칙

- [필수] 파일 Read 후에만 시뮬레이션
- [필수] "아마도" 금지 - 확인된 것만 서술
- [필수] 2회 루프 후 실패 시 사용자 승인
- [금지] 파일 읽기 없이 추론만으로 시뮬레이션
