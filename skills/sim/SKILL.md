# /sim - 시나리오 시뮬레이션

> "적용 후 시뮬레이션 돌려서 문제점 및 보완점 체크하고 수정 후 정리해줘"

## EazyCheck 내 위치

```
[3+ 파일 변경]
토론 (코드 전, 설계 검토) -> 코드 작성 -> sim (코드 후, 여기) -> Check (도구)

[수동 호출]
/sim -> 언제든 실행 가능 (파일 수 무관, 코드 전후 무관)
```

### 역할 분리

| 구분 | /sim | 토론 | Check |
|------|------|------|--------|
| 관점 | 시나리오 (미래 영향) | 설계 (구조 충돌) | 도구 (기술 결함) |
| 방식 | 탐색형 (모르는 문제) | 검토형 (설계 판단) | 체크리스트형 (알려진 패턴) |
| 시점 | 코드 후 (자동) / 언제든 (수동) | 코드 전 (3+ 파일) | 코드 후 (자동) |

## 자동 트리거

| 조건 | 동작 |
|------|------|
| 3+ 파일 변경 완료 | 토론 후 코드 작성 후 자동 실행 |
| `/sim` 직접 호출 | 전체 파이프라인 실행 |
| "시뮬레이션", "장기 테스트" | /sim 제안 |

### 생략 조건
- 단순 버그 수정 (1-2 파일)
- 스타일/포맷 수정
- 문서/테스트만 변경

## 6단계 파이프라인

### Phase 1: 현실 기반 고정
변경된 코드를 실제로 읽는다. 추측 금지.

```
- 변경된 파일을 실제로 Read
- 직접 의존성 파악
- 현재 상태를 사실로 확정
```

### Phase 2: 시나리오 생성
시간축: 즉시 / 1주 / 1개월 / 3개월+

```
각 시간축에서:
- 정상 동작 (happy path)
- 다른 부분과 충돌
- 전제가 무너지는 경우
- 규모 확대 (사용자 10배, 데이터 10배)
```

시나리오 최소 5개.

### Phase 3: 이중 평가 (CoVe + Self-Critique)

```
[CoVe 검증]
  1. 초안 분석
  2. "이것이 실제로 발생하는가?" 검증 질문
  3. 초안을 보지 않고 독립 답변 (편향 차단)
  4. 검증 통과한 항목만 최종 결과에 포함

[Self-Critique]
  - 시나리오가 현실적인가?
  - 놓친 시나리오가 있는가?
```

### Phase 4: 즉각 교정 (최대 2회 반복)

```
- high: 즉시 수정
- medium: 수정안 제시 후 승인 대기
- low: 보고만

수정 후 새 문제 발견 시 -> Phase 3으로 (최대 2회)
```

### Phase 5: 구조화 출력

```
[/sim 결과]
- 대상: {변경 내용}
- 시나리오: N개 (검증 통과만)
- 수정: N건
- 미래 감시 항목: ...
- 종합: OK / WARN / FAIL
```

### Phase 6: 기록
새로운 패턴 발견 시 patterns.json에 추가.

## 품질 규칙

- [필수] Phase 1에서 실제 파일 Read 후에만 Phase 2 진행
- [필수] 검증 실패 시나리오는 최종 보고에서 제외
- [필수] "아마도" 사용 금지 - 확인된 것만 서술
- [금지] 파일 읽기 없이 추론만으로 시나리오 생성
- [금지] Phase 3 건너뛰고 바로 수정
- [금지] 시나리오 3개 미만으로 보고 완료
