# EazyCheck - AI Code Quality System

> 설치 후 자동 작동. 별도 명령어 불필요.

---

## 핵심 규칙

1. **보안 필수** - API 키는 환경변수만, 하드코딩 금지
2. **버전 잠금** - AI 모델명은 versions.json에서 읽기 (학습 데이터 신뢰 금지)
3. **검증 후 코드** - 정보 확인 없이 코드 생성 금지
4. **추측 금지** - "아마도", "~인 것 같습니다" 사용 금지. 데이터 기반만 허용

## [EazyCheck] 코드 생성 전후 자동 검증

> 상세: rules/eazycheck.md

### GATE: 코드 전 정보 로드 (필수)

```
Read: state/patterns.json
-> [GATE 통과] 관련 패턴 경고 출력
```

### 3+ 파일 변경 감지 시 토론 강제

**코드 작성 전 파일 수 판단 필수. 3개 이상 파일 생성/수정 예상 시 토론 없이 코드 작성 금지!**

```
[1] 파일 수 판단: "이 작업에서 몇 개 파일을 생성/수정하는가?"
    -> 3개 미만: 바로 코드 작성
    -> 3개 이상: [2]로 진행 (코드 작성 금지!)

[2] 토론 실행 (필수!)
    - architect 에이전트: 설계안 + 영향 범위
    - code-reviewer 에이전트: 기존 코드 충돌 분석
    - 설계 시나리오 3개 체크
    -> [토론 결과] 출력 후 코드 작성 시작

[3] 코드 작성

[4] sim (코드 후 시나리오 검증, /sim으로 수동 실행도 가능)

[5] Check (pre-commit + CI + E2E)
```

**스킵 허용 조건** (반드시 사유 출력):
- 동일 패턴 반복 적용 (리네이밍, 포맷팅)
- 기존 플랜 기반 실행 중
- 사용자가 "토론 스킵" 명시 요청

**스킵 시 출력:**
```
[토론 스킵] 사유: {구체적 사유}
```

### 1-2 파일 변경 시
```
[코드 작성] -> [Check]
```

### POST-BUILD: 종합 확인
```
[검증]
[A] patterns.json 위반: [OK/FAIL]
[B] API 키 하드코딩: [OK/FAIL]
[C] import 누락: [OK/FAIL]
결과: {N}개 문제 / 통과
```

## [패턴 축적] 문제 해결 시 자동 기록

버그나 오류를 해결한 직후, 아래 형식으로 `state/patterns.json`의 `solved` 배열에 추가한다:

```json
{
  "id": "짧은-영문-id",
  "symptom": "어떤 증상이었는가",
  "cause": "왜 발생했는가",
  "solution": "어떻게 해결했는가",
  "frequency": "medium"
}
```

**트리거**: 에러 해결, 버그 수정, "됐어/해결됐다/고쳤어" 등 완료 표현 감지 시
**규칙**: 이미 같은 패턴이 있으면 건너뜀. 추가 후 `_meta.solved_count` 업데이트.

## [디버깅] 근본 원인 분석

### 트리거 키워드
"아직도", "똑같아", "그대론데", "여전히", "안 바뀌었어", "같은 에러", "또"

### 강제 규칙
- [금지] 코드 읽지 않고 수정
- [금지] 추측성 표현 ("아마도", "~인 것 같습니다")
- [필수] 코드 Read -> 원인 분석 -> 수정 제안 (순서 엄수)

## 검증 시스템 역할 분리

| 상황 | 시스템 | 시점 |
|------|--------|------|
| 코드 생성 전 | GATE (patterns.json 경고) | 자동 |
| 3+ 파일 설계 | 토론 (architect + code-reviewer) | 자동 |
| 코드 생성 후 시나리오 | sim (/sim 스킬) | 3+ 파일 자동 / 수동 |
| 코드 생성 후 도구 | Check (pre-commit, CI, E2E) | 자동 |
