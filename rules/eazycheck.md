# EazyCheck v5.0

> 역할 분리: 도구(Check) + 시나리오(sim) + 설계(토론)

---

## 설계 원칙

| 구분 | 담당 | 성격 |
|------|------|------|
| **GATE** | patterns.json 로드 | 코드 전 1회 |
| **토론** | architect + code-reviewer + 설계 시나리오 | 코드 전, 3+ 파일 |
| **Check** | pre-commit, CI, E2E (도구 전용) | 코드 후 자동 |
| **/sim** | 시나리오 시뮬레이션 | 코드 후, 3+ 파일 자동 / 수동 호출 가능 |

---

## 통합 흐름

```
[사용자 요청]
    |
    v
[GATE] patterns.json Read (1회)
    - 관련 패턴 경고 출력
    |
    v
[파일 수 판단]
    |
    +-- [1-2 파일] --> [코드 작성] --> [Check] --> [완료]
    |
    +-- [3+ 파일] --> [토론] --> [코드 작성] --> [sim] --> [Check] --> [완료]
```

---

## GATE: 코드 전 정보 로드

### 실행 (필수)

```
Read: state/patterns.json
```

### 출력

```
[GATE 통과]
- 패턴: solved {N}개, preventive {M}개, mistakes {K}개
- 관련 경고: {있으면 패턴ID + symptom + solution}

코드 생성을 시작합니다.
```

### 규칙
- 세션 내 1회만 실행 (반복 불필요)
- 관련 패턴 발견 시 경고 출력

---

## 토론 패턴 (3+ 파일, 코드 전) - 강제!

> **3+ 파일 생성/수정 시 토론 없이 코드 작성 금지!**

### 트리거
- 3+ 파일 변경 예상 OR 설계/리팩토링 키워드

### 스킵 허용 조건
- 동일 패턴 반복 적용 (리네이밍, 포맷팅)
- 기존 플랜 기반 실행 중
- 사용자가 "토론 스킵" 명시 요청
- 스킵 시 반드시 출력: `[토론 스킵] 사유: {구체적 사유}`

### 실행

```
[토론]
[1] architect (설계안 + 영향 범위)    -- 병렬 실행
[2] code-reviewer (기존 코드 충돌)    --

[3] 설계 시나리오 체크 (3개)
    - 기존 시스템 충돌 여부
    - 전제 유효성 (6개월 후)
    - 규모 확장성 (10배)

[4] 결과 병합 -> 최종 플랜
```

## Check: 도구 기반 검증 (코드 후)

### Check 1: Pre-commit Hooks
- Ruff: Python 린터/포매터
- check-ast: AST 검증
- 하드코딩 탐지 (API 키, 확장자)

### Check 2: GitHub Actions
- AI 자동 코드 리뷰
- 보안 스캔 (Bandit)
- 영향 분석

### Check 3: E2E 통합 테스트
- 실제 동작 검증

## /sim: 시나리오 시뮬레이션 (코드 후)

> skills/sim/SKILL.md 참조

### 자동 트리거 (3+ 파일 변경)
- 코드 작성 완료 후, Check 도구 실행 전에 실행
- 실제 코드를 Read하여 시나리오 생성

### 수동 호출
- `/sim` 으로 언제든 호출 가능
